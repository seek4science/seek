# Plan: Migrate from DelayedJob to Solid Queue (#2656)

Issue: [#2656 Investigate using SolidQueue for managing the job queue](https://github.com/seek4science/seek/issues/2656)
Branch: `solid-queue-2656`

## Goal

Replace `delayed_job_active_record` with `solid_queue` as the `ActiveJob` backend, without changing any job class behaviour beyond what's required for parity, and without regressing the multi-queue setup SEEK currently relies on.

## Current state (for reference)

- `config/application.rb:78` — `config.active_job.queue_adapter = :delayed_job`. All async work already goes through `ActiveJob` (`app/jobs/`, 44 job classes), not the DelayedJob API directly.
- `config/initializers/delayed_job_config.rb` — worker tuning (`max_attempts = 1`, `max_run_time = 1.day`, `sleep_delay`, `reserve_sql_strategy = :default_sql` to avoid deadlocks).
- `app/jobs/queue_names.rb` — 8 named queues (`default`, `mailers`, `authlookup`, `remotecontent`, `samples`, `indexing`, `templates`, `datafiles`).
- `lib/seek/workers.rb` — computes the *active* subset of those queues at boot from `Seek::Config` feature flags (e.g. `AUTH_LOOKUP` queue only runs if `Seek::Config.auth_lookup_enabled`), then daemonizes one single-threaded `delayed_job` worker process per active queue.
- `docker/start_workers.sh`, `script/check_worker_pids.sh`, `rake seek:workers:start/stop` — process management and Docker healthcheck, all delayed_job-specific (parse `tmp/pids/delayed_job.*.pid`).
- `config/schedule.rb` (via `whenever` gem) — cron-driven scheduling. Some entries just call `.perform_later`/`.queue_job` on a job (candidates for Solid Queue's recurring jobs), others run rake tasks, shell commands, or plain class methods that aren't jobs at all (`Galaxy::ToolMap.instance.refresh`, `kill-long-running-soffice.sh`, `sitemap:refresh`, `db:sessions:batch_trim`) — these can't move to Solid Queue and will stay on `whenever`/cron.
- `app/jobs/application_job.rb` — custom `queue_job`/`follow_on_job?`/duplicate-avoidance logic layered on top of `enqueue`, plus manual exception reporting via `Seek::Errors::ExceptionForwarder` (no automatic retries: `max_attempts = 1`).
- Test env uses `queue_adapter = :test`; unit tests optionally run against SQLite (`database.github.sqlite3.yml`) as well as MySQL. The `pg` gem is also already in the `Gemfile`, so Postgres is a supported adapter alongside `mysql2` and `sqlite3`.
- Admin UI surfaces delayed_job internals directly: `app/views/admin/_restart_buttons.html.erb` shows expected vs. running worker processes (`Seek::Workers.active_queues.count`, `Seek::Util.delayed_job_pids`) and a "Restart background job workers" button (`restart_delayed_job_admin_path` → `rake seek:workers:restart`); `app/views/admin/stats/_job_queue.html.erb` lists `Delayed::Job` rows (priority, queue, attempts, run/locked/failed times, handler, last error) with a "Clear failed jobs" button (`AdminController#clear_failed_jobs` → `Delayed::Job.where('failed_at IS NOT NULL').destroy_all`).

## Key risks identified

1. **Concurrency model change**: DelayedJob workers are separate single-threaded OS processes; Solid Queue's default worker runs multiple jobs concurrently via threads in one process. **Phase 0 audit result (see below)**: the real risk isn't Rugged/git, HTTP clients, or Sunspot (all confirmed safe — fresh instances per call, or already thread-local), it's two pieces of **process-global mutable state** that were only safe under DelayedJob because each worker was a separate OS process:
   - `User.current_user` is a `cattr_accessor` (`app/models/user.rb:64`, a true class variable), set via `User.with_current_user` in `OpenbisSyncJob`, `FairDataStationImportJob`, `FairDataStationUpdateJob`, `SampleDataPersistJob`, `UnzipDataFilePersistJob`. Two such jobs running concurrently on different threads can clobber each other's current user (affects authorization, `contributor`, activity logging, and `ExceptionForwarder`'s default user).
   - `$authorization_checks_disabled` is a plain Ruby global (`lib/extensions/object.rb:4-9`), toggled by `disable_authorization_checks`, used in `LinkingSamplesUpdateJob`, `RemoteGitContentFetchingJob`, `SamplesBatchDeleteJob`, `Set/RemoveSubscriptionsForItemJob`, `LifeMonitorStatusJob`. Concurrent jobs can re-enable/disable authorization checks mid-flight for one another.

   Note both globals are already shared across Puma's 5 web-server threads (`config/puma.rb`) today, so this is a pre-existing latent bug class — Solid Queue just newly exposes it to job execution too, at higher likelihood given jobs hold these flags for longer stretches than a web request. **Mitigation for this migration: keep 1 thread per queue initially (matching current single-worker-per-queue behaviour), so no two jobs from the same or different queues execute in the same process concurrently.** Making these genuinely thread-safe (e.g. `Concurrent::ThreadLocalVar`/`RequestStore`) is worth doing but is a separate piece of work, not a blocker for this migration as long as concurrency stays effectively serial.
   - Secondary, lower-probability finding: `Git::Repository#git_base` correctly memoizes a fresh `Rugged::Repository` per AR instance (not shared), but two *different* jobs concurrently operating on the *same underlying git repo id* on separate threads (e.g. `RemoteGitFetchJob` and `RemoteGitContentFetchingJob` for the same repo) each open their own `Rugged::Repository` for the same on-disk path — worth keeping in mind if queue concurrency is increased later, though not currently known to be triggered by any existing code path.
2. **`config/schedule.rb` only partially migrates** — RESOLVED: everything migratable moved to `config/recurring.yml`, leaving only the Docker-only `kill-long-running-soffice.sh` reaper, which was then moved to a static `docker/seek.crontab` fed to supercronic (see Phase 6). The `whenever` gem and `config/schedule.rb` have been removed; only genuine OS-level process reaping remains on cron.
3. **Dynamic queue activation**: `Seek::Workers.active_queues` reads `Seek::Config` feature flags at boot. Solid Queue's `config/queue.yml` is static (ERB+YAML, evaluated once at supervisor boot), so replicating "only run a worker for enabled features" requires DB access at Solid Queue supervisor boot and a restart when a feature flag changes — same category of concern as the recent `settings_table_available?` fallback work.
4. **Retry semantics**: current `max_attempts = 1` (fail fast + manual exception report) must be deliberately replicated (`retry_on`/`discard_on`), otherwise jobs may start silently retrying (risk: duplicate emails).
5. **Cutover of in-flight jobs**: rows already in `delayed_jobs` don't automatically appear in Solid Queue's tables. Decision: cutover (flipping the adapter and deploying) is **decoupled** from migrating those rows — the adapter flip happens first, with Solid Queue simply starting from an empty queue, and migrating the pre-existing `delayed_jobs` rows into Solid Queue's tables is handled as a separate, later step (see Phase 5). This means there's a window, potentially spanning a release or more, where any rows still sitting in `delayed_jobs` at cutover time are not being processed by anything. `delayed_job_active_record` and the `delayed_jobs` table must stay in the app (gem + schema) until well after that migration has run.
6. **Infra rewrite surface** — RESOLVED in Phase 3: `lib/seek/workers.rb`, `docker/start_workers.sh`, `script/check_worker_pids.sh`, `rake seek:workers:*` rewritten/removed against Solid Queue's process model (`bin/jobs` via `script/run_solid_queue.sh`). Also found and fixed two more delayed_job-specific spots not originally listed here: the non-Docker deployment scripts (`script/update-from-git.sh`, `script/mini-update-from-git.sh`) and `ApplicationStatus#refresh`/the `/application_status` text endpoint.
7. **Admin UI is delayed_job-specific** — RESOLVED in Phase 3: `_restart_buttons.html.erb` and `stats/_job_queue.html.erb` now show Solid Queue's supervisor status and `SolidQueue::Job` rows respectively, so they reflect real state rather than going stale.

## Phased plan

### Phase 0 — Audit & decisions
- [x] Audit job classes for thread-safety. Result: no issues found with Rugged/git, HTTP clients, Sunspot, or image processing (all use fresh instances per call or are already thread-local). The real finding is two process-global mutable state points (`User.current_user`, `$authorization_checks_disabled`) — see risk #1 above for detail and mitigation (1 thread per queue for this migration; making these properly thread-local is separate follow-up work).
- [x] Database: Solid Queue tables will live in the **shared** app database for now, not a separate database — no changes needed to `config/database.yml`'s single-database structure at this stage. A dedicated queue database can be revisited later if isolation becomes necessary.
- [x] Must support all three database adapters SEEK already supports: **MySQL** (`mysql2`, primary), **SQLite** (`sqlite3`, test/dev), and **Postgres** (`pg`, already in the `Gemfile`) — Solid Queue supports all three, so this constrains testing/verification (Phase 2) to cover all three, but doesn't rule anything out.
- [x] Topology: replicate today's "one queue per active feature flag" 1:1 in `config/queue.yml` for this migration, rather than consolidating. Job classes only declare a queue *name* via `queue_as QueueNames::X`; they have no dependency on how many worker processes/threads service that queue. Consolidating (e.g. wildcard queue matching, grouping queues into shared worker processes) is therefore a config-only change with no job code changes or data migration required, so it can be revisited later as a pure ops optimization without cost to deciding 1:1 now.
- [x] Confirm Rails/Ruby versions already satisfy Solid Queue's requirements: Ruby `3.3.10` (`.ruby-version`), Rails `8.1.3` (`Gemfile.lock`) — both comfortably above Solid Queue's minimums (Rails 7.1+). `solid_queue` is not yet in `Gemfile.lock`, so it needs to be added explicitly in Phase 1 (Rails 8 ships it as the *default* for new apps, but SEEK's existing app doesn't have it wired up yet). Separately noted: `AGENTS.md` still describes the app as "Rails 7.2" — worth a docs fix, but unrelated to this migration.

- [x] **Bug fix (found via CI after Phase 4)**: `db/schema.rb`'s table order is alphabetical (matches how `ActiveRecord::SchemaDumper`/the `solid_queue` install generator's own template both sort tables), which put `solid_queue_jobs` *before* three tables that hold a foreign key referencing it (`solid_queue_ready_executions`, `solid_queue_recurring_executions`, `solid_queue_scheduled_executions`). MySQL's `DROP TABLE ... CASCADE` doesn't actually cascade (the keyword is accepted but ignored), so re-running `db:schema:load`/`db:setup` against a MySQL database that already has the schema loaded failed with `Cannot drop table 'solid_queue_jobs' referenced by a foreign key constraint` (CI's `rake db:setup` job, which loads the schema once via a separate step and then again via `db:setup` itself). Fixed by relocating `solid_queue_jobs`'s `create_table` block to after all six tables that FK-reference it (still before `solid_queue_semaphores`), with a comment explaining why it's out of alphabetical order. Verified locally against real MySQL: loaded the schema twice in a row into a scratch database (reproducing the CI failure beforehand, confirming the fix afterward), then ran `rake db:setup` end-to-end successfully.
- [x] `config/queue.yml`: one worker per queue, ERB-gated by the same `Seek::Config` flags as `Seek::Workers.active_queues` (mirrors 1:1 topology decision), **1 thread per queue** per the Phase 0 mitigation. Verified it parses via `ActiveSupport::ConfigurationFile` and only emits workers for currently-enabled features.
- [x] `config/recurring.yml`: covers the six job-enqueue entries from `config/schedule.rb`. Schedules were derived from the actual output of `bundle exec whenever` (not reasoned about in the abstract) to match today's real cron times exactly, including whenever's quirk of ignoring the `at:` offset for hour-frequency jobs (`RegularMaintenanceJob`, `AuthLookupMaintenanceJob`) while honouring it for day-frequency ones (`PeriodicSubscriptionEmailJob`, `LifeMonitorStatusJob`). All schedules verified to parse as valid cron via `Fugit`. Non-job entries (rake tasks, shell commands, `Galaxy::ToolMap.instance.refresh`, `ApplicationStatus.instance.refresh`, `Seek::BioSchema::DataDump.generate_dumps`) intentionally left out, staying on `whenever`/cron as decided in Phase 0/risk #2.
- [x] **Bug fix (found after Phase 4)**: two separate `config/schedule.rb` problems surfaced together. (1) The Phase 1 migration to `config/recurring.yml` never actually *removed* the migrated entries from `config/schedule.rb` — every job-enqueue entry (`PeriodicSubscriptionEmailJob`, `RegularMaintenanceJob`, `AuthLookupMaintenanceJob`, `LifeMonitorStatusJob`, `NewsFeedRefreshJob`, `ApplicationJob.queue_timed_jobs`) was still present in both files, meaning every deploy that runs `bundle exec whenever --update-crontab` (as `script/update-from-git.sh`/`script/mini-update-from-git.sh`/Docker already do) would double-enqueue all of them: once via cron, once via Solid Queue's own scheduler. Fixed by removing all six from `config/schedule.rb`, leaving only the rake tasks (`sitemap:refresh`, `db:sessions:batch_trim`) and the Docker-only shell command (`kill-long-running-soffice.sh`), which don't map onto `recurring.yml`'s mechanisms. (2) The earlier assumption that "non-job" entries couldn't move to `recurring.yml` was wrong: `SolidQueue::RecurringJob#perform` just `eval`s the configured `command:` string, so it isn't limited to `ActiveJob` calls (`clear_finished_jobs` already relied on this). `Galaxy::ToolMap.instance.refresh`, `ApplicationStatus.instance.refresh`, and `Seek::BioSchema::DataDump.generate_dumps` were moved over as `command:` entries on this basis, leaving only the genuinely non-Ruby rake/shell entries on `whenever`/cron.
- [x] **Second bug fix, found while fixing the first**: any `command:`-only `recurring.yml` entry (no `class:`) enqueues via `SolidQueue::RecurringJob`, whose own `queue_as` is `solid_queue_recurring` — a queue `config/queue.yml` never had a worker for, since it only accounted for SEEK's own `QueueNames`. This meant `clear_finished_jobs` and `queue_timed_jobs` (both already committed in Phase 1/3) had never actually run: verified by enqueuing a scratch command-based recurring task against the real, already-running dev supervisor and confirming it sat unprocessed. Fixed by adding an always-on (not feature-gated) `solid_queue_recurring` worker to `config/queue.yml`; re-verified the same scratch task then completed normally. Covered by `test/integration/recurring_test.rb`'s "queue that has a configured worker" test, which fails without this fix (confirmed by temporarily reverting it).
- [x] Added `test/integration/recurring_test.rb`: verifies every `config/recurring.yml` entry's exact class/command/args/schedule, that all schedules parse as valid cron, that every entry resolves to a valid `SolidQueue::RecurringTask` whose target queue has a worker configured (the regression test for the bug above), the offset/config-driven schedule changes, and an end-to-end execution test (via `SolidQueue::RecurringTask#enqueue` + `perform_enqueued_jobs`, the same code path Solid Queue uses in production) for every entry. `test/integration/schedule_test.rb` was rewritten down to match the trimmed `config/schedule.rb` (just the two rake tasks and the Docker-gated shell command), with an explicit assertion that no `runner` jobs remain, to catch a repeat of the double-scheduling bug.
- [x] **Polling interval tuning**: `config/queue.yml`'s workers originally inherited Solid Queue's own gem-level default `polling_interval` (0.1s). That default assumes the stock topology of one worker polling all queues via a wildcard; our one-worker-per-queue topology (Phase 0 mitigation, see risk #1) multiplies it into up to 9 separate continuous polling loops - up to ~90 poll queries/second against the DB even when idle. Raised to 1s for all queues, except `AUTH_LOOKUP` (0.5s, the highest-frequency/most latency-sensitive queue), trading a small amount of job-pickup latency for a large reduction in idle DB polling load. Does not change the topology or thread-safety mitigation itself - purely a per-worker tuning value.
- [x] Retry/failure behaviour: **no code change needed**. Verified empirically (`TmpFailJob.perform_now` raising inside `perform`) that `ApplicationJob`'s `rescue_from(Exception)` (`app/jobs/application_job.rb`) swallows the exception inside `ActiveJob::Base#perform_now` itself — this happens before the queue adapter ever sees an error, so it's already adapter-agnostic. The exception never propagates to Solid Queue, meaning `solid_queue_failed_executions` will stay empty for these jobs, and no `retry_on`/`discard_on` declarations are needed to preserve the current fail-fast + exception-forwarding behaviour. (Side note for Phase 3: this also means the admin "Clear failed jobs" UI is unlikely to ever have anything to clear once Solid Queue is live, since it mirrors current behaviour where `Delayed::Job.failed_at` is essentially never set either.)

### Phase 5 — Migrate historical `delayed_jobs` rows (later, separate release)
- [x] Migration logic lives in `Seek::DelayedJobMigrator` (`lib/seek/delayed_job_migrator.rb`), which reads all remaining rows from `delayed_jobs` (left over from before the Phase 4 cutover) and re-enqueues equivalent jobs into Solid Queue (mapping queue, `run_at`, priority, attempts). Locked rows (`locked_at`/`locked_by` set) are migrated like any other pending row — the lock is meaningless once the delayed_job workers are gone. Already-failed rows (`failed_at` set) are deleted rather than migrated. Queued reindexing jobs (`ReindexAllJob`/`ReindexingJob`, the only two on the `indexing` queue) are also dropped rather than migrated, and the `ReindexingQueue` table (which `ReindexingJob` pulls its batches from) is emptied: the upgrade runs a full `seek:reindex_all` immediately afterwards, re-enqueuing a complete reindex of every searchable type, so any reindex work queued before the cutover is redundant. **Key implementation decision:** each Solid Queue row is built *directly from the raw ActiveJob payload* (`dj.payload_object.job_data`) via `SolidQueue::Job.create!`, **not** by deserialising the job and calling `.enqueue`/`.serialize`. Re-serialising would force resolution of the job's GlobalID arguments, raising `ActiveJob::DeserializationError` — and aborting the whole migration — for any job whose referenced record has since been deleted. Building the row directly keeps the original serialised arguments intact; deserialisation then happens lazily at perform time, where `ApplicationJob`'s `rescue_from` already swallows a missing-record `DeserializationError`. The delayed_job `attempts` count is folded into ActiveJob's `executions`. `SolidQueue::Job`'s own `after_create :prepare_for_execution` then correctly makes past-due jobs ready and keeps future-scheduled ones scheduled. Verified (SEEK enqueues *only* via ActiveJob — no `.delay`/`handle_asynchronously`/plain `Delayed::Job.enqueue` — so every handler is an `ActiveJob::QueueAdapters::DelayedJobAdapter::JobWrapper`). Covered by `test/unit/delayed_job_migrator_test.rb` (queue/priority/run_at preservation, ready-vs-scheduled dispatch, attempts→executions, failed-row deletion, reindex-job dropping, ReindexingQueue clearing, deleted-argument-record robustness, empty-table no-op).

## Open questions

- Target thread pool sizes per queue — not something to decide up front; determined by the outcome of the Phase 0 thread-safety audit (start conservative, i.e. 1 thread per queue matching current behaviour, for anything the audit doesn't clear for higher concurrency).

## Rollback plan

Keep `queue_adapter` a one-line config change through Phase 5. `delayed_job_active_record`, its tables, and the old worker scripts remain installed (not removed until Phase 6 in a later version), so if issues surface after Phase 4 the adapter can be flipped back to `:delayed_job` while investigating, without needing to reinstall anything. Since Phase 4 and Phase 5 are decoupled, a rollback after Phase 4 but before Phase 5 simply resumes delayed_job workers against the (still-intact) `delayed_jobs` table plus whatever accumulated there in the meantime.
